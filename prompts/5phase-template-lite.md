## 一、核心原则

写 prompt 时真正起作用的不是段落数量，而是三件事：

1. **可判定的目标**。不写"改对""搞清楚"这种无法验收的说法，写"改完后进程能正常启动""产出文档说明 X 字段的解析路径"这种完成与否一目了然的结果。
2. **素材给路径、给原文，不给转述**。文件/目录用 `@路径` 直接标出，让 Agent 一步定位；评审意见、报错日志等原始材料整段粘贴——转述会丢掉"谁在哪一行说了什么"这类逐条对应修改点时必需的信息。
3. **只写非默认的要求**。Agent 默认就会读代码、会汇报结果，这些不用写。要写的是它默认不会做的事：已知的偏差来源、任务边界之外还要不要扫描同类问题、改文件前要不要停下来等确认。

## 二、模板正文

必填两段（`目标` `素材`），其余段落有内容才加，没有就整段删掉（这一点与旧模板"缺省写无"的约定相反——空段落只是噪音）。

```markdown
## 用户目标

一两句话说清：在什么场景下（迁移/排查/评审落地/调研...），要达成什么可判定的结果。

## 参考背景

- 目标对象：`@具体路径`（要改/要分析的文件、目录、配置库）
- 参考资料：`@路径`（repo-wiki、以往调研文档、其他产品线已验证的样例）
- 已知信息：用户已经知道的偏差来源、踩过的坑、确定不用查的方向
- 原始材料：评审意见/报错日志/问题清单，整段粘贴原文

## 任务列表
> 可选——目标一句话说不完时才需要

1. 动作一（每条是一个可核验的产出，不是模糊意图）
2. 动作二
- 边界说明：任务是否止步于上述清单（例如"评审意见可能没覆盖全部同类问题，需全量扫描"要显式写出，否则 Agent 只会逐条对应）
- 分工说明：哪些由 Agent 完成、哪些由用户手动执行（如"实际测试由用户进行，你只需产出操作文档"）

## 交付
> 可选——对产出物有格式/位置要求时才需要

- 文档保存路径、命名规范
- 内容要求：是否需要原理性说明（可复用）而不只是操作记录（一次性）；超出任务范围的发现是否单独分节

## 约束
> 可选——从下方片段库按需取用，或写任务特有的红线

## 检查点
> 可选——改文件、动线上环境等不可逆动作前需要人工确认时才需要

```

## 三、常用约束片段库

以下片段在多次实际任务中被验证有效，按需复制到「约束/检查点」段，不要默认全带上——每条约束都有注意力成本，带三条精准的比带八条泛泛的效果好。

**查证优先（适合配置驱动、反射查字段类型的系统）**

> Prefer tools or source code read than assumption. 一切"为什么失败""字段该怎么配"的结论必须先读源码/配置样例/日志验证，不能凭经验猜测。字段名相近但 proto 里不存在这类问题必须逐项核实。

**并行调研、串行执行（适合调研面广的任务）**

> 调研面广时拆给后台 agent 并行调研；但修改文件的执行动作必须在方案确认后单线程收敛进行，不允许并行修改。

**不确定性标注（适合结论要被后续任务复用的调研）**

> 未经直接验证的结论必须显式标注推断依据和置信度，不能与已验证事实混写。查不到直接证据就如实说没查到。

**先规划后执行（适合有修改动作的任务）**

> 任何修改文件的动作，先给出完整方案（改动清单+每条的依据+diff）等用户确认再落地。方案中"不影响正确性但涉及风格取舍"的项、以及调研中发现的原始材料未覆盖的同类问题，都要在方案阶段作为独立问题呈现，不能自行拍板或做完才提。

**脚本复核（适合正确性条件可编程验证的任务）**

> 执行完成后，可编程校验的正确性条件（必填字段全覆盖、无重复映射等）必须用脚本跑一遍复核，把结果作为完成汇报的一部分，不能凭读完文本的印象下结论。

## 四、LLM 优化指南：如何依据本模板优化一个 prompt

当用户给出一个粗糙的 prompt 并要求依据本模板优化时，按以下步骤操作。

**第 1 步：判定任务类型，决定段落取舍**

- 纯调研/分析任务 → 目标 + 素材，通常够了；结论会被复用则加「不确定性标注」约束
- 修改配置/代码任务 → 目标 + 素材 + 任务 + 「先规划后执行」检查点；能脚本验证的加「脚本复核」
- 产出文档任务 → 目标 + 素材 + 交付（明确保存路径和"原理性说明"要求）
- 混合任务 → 按上述叠加，但警惕段落膨胀：如果优化后的 prompt 比原始版本长两倍以上，先检查是不是把默认行为也写进去了

**第 2 步：逐项体检原始 prompt，发现缺陷就修**

| 检查项 | 缺陷特征 | 修法 |
|--------|----------|------|
| 目标可判定性 | "优化一下""确认没问题" | 改写为可验收的终态："启动检查通过""产出 X 文档回答 Y 问题" |
| 素材可定位性 | "那个点击配置文件""上次的调研文档" | 向用户要确切路径，写成 `@路径`；找不到就明确标注"路径待用户补充" |
| 原始材料完整性 | 转述评审意见/报错（"评审说字段名不对"） | 要求粘贴原文，保留评论人、位置、原话 |
| 已知信息前置 | 用户已知的偏差没写出来 | 追问"你已经知道哪些原因/排除了哪些方向"，写进素材段——省下 Agent 从零重新发现的轮次 |
| 任务边界 | 没说清是否要扫描清单外的同类问题 | 显式写"止步于清单"或"需全量扫描"二选一 |
| 人机分工 | 没说清哪些步骤用户手动做 | 显式写"测试由用户手动进行，你只产出文档"这类分工 |
| 约束冗余 | 堆了一堆与任务无关的通用约束 | 只保留与本任务风险点匹配的片段（见第三节的适用场景标注） |
| 检查点缺失 | 有改文件动作但没有确认环节 | 加「先规划后执行」 |

**第 3 步：输出**

- 按第二节的段落结构输出优化后的 prompt，空段落删除
- 若体检中发现信息缺口（路径、原文、已知信息）无法自行补齐，在优化稿末尾列出"待用户补充"清单，而不是编造占位内容
