# Prompt Template

用于向 Coding Agent（Kiro/Claude Code 等）下达"基于某种输入材料（评审意见、问题描述、需求、异常日志等），对代码库中的配置/代码进行核实并修改"类任务的标准结构。它不绑定特定场景——只要任务形态是"先查证据、再出方案、确认后动手、动手后复核"，都可以套用这 5 段。

典型适用场景举例：评审意见落地、跨模块/跨仓库行为对齐、故障排查后的修复、依赖升级带来的适配、重构过程中的偏差修正等。

## 使用方法

复制下面的 5 段结构，填入具体任务内容。5 段全部保留，缺省内容写"无"而不是删除该段——保留结构本身就是在提醒 Agent 该走哪些步骤。

---

## 模板正文

```markdown
### User Request & Target

需要完成的、明确的动作列表。用有序或无序列表列出，每条动作是一个可核验的产出，不是模糊的意图。通常包含：

1. 需要修改的具体文件/对象（给出确切路径）
2. 修改依据（评审意见/问题清单/需求描述/异常日志等原始材料，直接粘贴或引用，不要转述——转述会丢信息，尤其是像"谁在哪一行提了什么意见""日志里具体报了什么"这种需要逐条对应修改点的场景）
3. 需要产出的说明性文档，及其保存路径
4. 如果任务边界不止输入材料本身（例如"评审意见可能没有覆盖全部同类问题，需要全量核查""异常日志只暴露了一个触发点，同类调用点都要排查"），要明确写出来，否则 Agent 默认只会逐条对应输入材料，不会主动扫描同类问题

### Context

用户当前所处的场景、目标对象、期望达成的结果。要包含：

- 正在做什么工程活动（评审落地/故障排查/重构/升级适配/行为对齐 等）
- 目标对象是谁：具体的文件、模块、配置库路径（用 `@路径` 或代码库引用标出，让 Agent 能直接定位，不必先花一轮去找）
- 对象当前处于什么状态、和什么东西做过对齐（如："A 模块的行为已确认，但 B 模块在特定条件下存在已知差异"）——明确指出"已知的偏差来源"，避免 Agent 从零开始猜
- 可供参考的同类/权威样例（其他模块的已验证实现、线上配置、规范文档等）路径
- 背景资料位置（repo-wiki、以往调研文档、agent memory 等）
- 期望达成的最终结果是什么（不是"改对"这种抽象说法，而是"这份代码/配置按新规则改完之后能正常启动/通过评审/行为与参照等价"这种可判定的结果）

### OutputFormat

对最终产出的结构与内容要求。至少覆盖：

- 修改说明文档要包含什么：每条修改的问题描述、代码依据（文件路径:行号）、修改前后对比（diff 形式）
- 是否需要解释背后的原理机制（如"相关模块是如何处理这种情况的"），而不只是"改了什么"——原理性说明能让文档在后续同类任务中被复用，而不是一次性的操作记录
- 文档保存位置、命名规范（如果项目已有归档规范，注明）
- 如果发现了原始材料未覆盖的问题，要求在文档中单独分节列出，避免和"任务要求的修改"混在一起造成误导

### Constraints

- **Prefer tools or source code read than assumption.** 一切关于"为什么会失败""应该怎么改"的结论，必须先读源码/配置样例/文档/日志验证，不能凭经验或常识猜测。任何代码库都容易出现"看起来合理但实际不对"的判断（比如参数名/类型相近但接口里根本不存在、函数有未声明的副作用、依赖版本之间行为有差异），必须逐项核实。
- **鼓励使用 workflow 或 agent 协作模式**：当调研面较广（需要同时读多个模块的源码、多份配置、多份文档）时，应拆分给后台 agent 并行调研，主线程同步做另一部分验证或规划，而非串行地一件件查。调研类任务和执行类任务（修改文件）要分开——调研可以并行探索，执行必须在方案确认后单线程进行。
- **不确定的内容必须显式标注**，不能用肯定语气陈述未经验证的结论。如果某个结论只能间接推断（比如"未在本仓库找到直接证据，但从命名和上下文推测应为..."），要写明推断依据和置信度，不要和已验证的事实混在一起。最终方案或选择交由用户进行确认或进行额外补充说明。

### Checkpoint

- 在得出结论、给出修改方案后，必须先暂停并把方案完整呈现给用户（表格/清单形式，标明每条的依据），不能直接执行修改。
- 如果方案中存在"是否要一并处理"这类不影响正确性但涉及风格/规范取舍的问题，要作为独立问题列出，等用户明确选择，不能自行决定后再告知。
- 如果调研过程中发现原始材料（评审意见/问题清单/需求/日志）之外的同类问题或遗漏项，同样要在方案阶段一并呈现，而不是执行完后才提及，给用户完整的决策信息。
- 执行完成后，如涉及可编程校验的正确性条件（如"所有必填字段都已配置""无重复映射""测试全部通过"），应当用脚本/工具跑一遍复核，将复核结果（通过/不通过+具体项）作为完成汇报的一部分，而不是仅凭改完的文本判断。
```

## 设计说明（为什么是这个结构）

这个模板背后对应的是一套"先查证据、后出方案、留检查点、再动手、再复核"的处理节奏。把任务拆成固定的 5 段并不是形式主义，而是每段都在堵一个 Agent 常见的失败模式：

1. **Context 段解决"Agent 从哪里开始查"的问题**：如果不给已知的偏差来源和参照样例，Agent 会从零假设，浪费大量轮次重新发现用户早就知道的背景。明确给出"已经确认了什么、已知哪里有偏差"，相当于把 Agent 的起点从空白挪到用户认知的前沿。
2. **Request 段的关键是"原始材料要给全量，不要转述"**：评审意见、异常日志这类材料往往需要逐条对应到具体的修改点，转述会丢失原始的措辞和上下文（谁提的、在哪一行提的、日志具体报了什么），转述后 Agent 可能会误解问题的准确指向。任务边界也要显式写出，否则 Agent 默认只处理字面提到的内容，不会主动扫描同类问题。
3. **OutputFormat 段要求"原理说明"而不只是"改了什么"**：一次性的操作记录用完就丢，但如果文档里写清楚了"为什么这块逻辑会在这种情况下出问题"，这份文档就能被下一次同类任务复用，减少重复调研成本。
4. **Constraints 段的核心是治理"看似合理但没验证"的风险**：任何代码库里，"读起来很像对的"和"实际是对的"之间都可能有鸿沟（参数名/类型相近、未声明的副作用、版本间行为差异），这类隐蔽问题只能靠逐项的源码/文档/样例核对，而不能靠经验判断。同时要求调研和执行分离——调研阶段允许并行、允许犯错重来，执行阶段（改文件）必须收敛到确认后的单一方案再动手。
5. **Checkpoint 段是这套流程里最容易被跳过、但最重要的一环**：如果 Agent 在方案阶段发现了"用户没问但同样存在的问题"，必须把这类发现和"用户明确要求的修改"分开呈现，并且对"要不要一并处理"这种没有唯一正确答案的取舍问题，显式提交给用户选择，而不是自己拍板。修改完成后，如果正确性条件可以用脚本验证，必须真的跑一遍，而不是靠读代码后的印象下结论。
