# Prompt Template：代码库配置/代码变更任务

用于向 Coding Agent（Kiro/Claude Code 等）提出"基于评审意见或问题描述，对代码库中的配置/代码进行核实并修改"类任务的标准结构。适用场景：多产品线迁移配置修正、CR 评审意见落地、代码库间行为差异排查后的修复等。

## 使用方法

复制下面的 5 段结构，填入具体任务内容。5 段全部保留，缺省内容写"无"而不是删除该段——保留结构本身就是在提醒 Agent 该走哪些步骤。

---

## 模板正文

```markdown
### Context

用户当前所处的场景、目标对象、期望达成的结果。要包含：

- 正在做什么工程活动（迁移/评审落地/故障排查/重构 等）
- 目标对象是谁：具体的文件、模块、配置库路径（用 `@路径` 或代码库引用标出，让 Agent 能直接定位，不必先花一轮去找）
- 对象当前处于什么状态、和什么东西做过对齐（如："已完全对齐旧系统 X 的配置，但新系统 Y 的字段解析形式发生了变化"）——明确指出"已知的偏差来源"，避免 Agent 从零开始猜
- 可供参考的同类/权威样例（其他产品线的线上配置、其他已验证过的实现）路径
- 背景资料位置（repo-wiki、以往调研文档、agent memory 等）
- 期望达成的最终结果是什么（不是"改对"这种抽象说法，而是"这份配置文件按新系统规则改完之后能正常启动/通过评审/行为与旧系统等价"这种可判定的结果）

### Request

需要完成的、明确的动作列表。用有序或无序列表列出，每条动作是一个可核验的产出，不是模糊的意图。通常包含：

1. 需要修改的具体文件/对象（给出确切路径）
2. 修改依据（评审意见/问题清单/异常日志等原始材料，直接粘贴或引用，不要转述——转述会丢信息，尤其是像"哪个人在哪一行提了什么意见"这种需要逐条对应修改点的场景）
3. 需要产出的说明性文档，及其保存路径
4. 如果任务边界不止评审意见本身（例如"评审意见可能没有覆盖全部同类问题，需要全量核查"），要明确写出来，否则 Agent 默认只会逐条对应评审意见，不会主动扫描同类问题

### OutputFormat

对最终产出的结构与内容要求。至少覆盖：

- 修改说明文档要包含什么：每条修改的问题描述、代码依据（文件路径:行号）、修改前后对比（diff 形式）
- 是否需要解释背后的原理机制（如"配置解析器是如何处理这个字段的"），而不只是"改了什么"——原理性说明能让文档在后续同类任务中被复用，而不是一次性的操作记录
- 文档保存位置、命名规范（如果项目已有归档规范，注明）
- 如果发现了原始材料未覆盖的问题，要求在文档中单独分节列出，避免和"任务要求的修改"混在一起造成误导

### Constraints

- **Prefer tools or source code read than assumption.** 一切关于"为什么会失败""字段该怎么配"的结论，必须先读源码/配置样例/文档验证，不能凭经验或常识猜测。配置驱动的系统尤其容易"看起来合理但实际不对"（比如字段名相近但proto里根本不存在），必须逐字段核实。
- **鼓励使用 workflow 或 agent 协作模式**：当调研面较广（需要同时读多个模块的源码、多份配置、多份文档）时，应拆分给后台 agent 并行调研，主线程同步做另一部分验证或规划，而非串行地一件件查。调研类任务和执行类任务（修改文件）要分开——调研可以并行探索，执行必须在方案确认后单线程进行。
- **不确定的内容必须显式标注**，不能用肯定语气陈述未经验证的结论。如果某个结论只能间接推断（比如"未在本仓库找到直接证据，但从命名和上下文推测应为..."），要写明推断依据和置信度，不要和已验证的事实混在一起。
- 先规划，后执行：任何会修改文件的动作，必须先给出完整方案（改动清单+依据+对不确定项的问题）并让用户确认，再落地。

### Checkpoint

- 在得出结论、给出修改方案后，必须先暂停并把方案完整呈现给用户（表格/清单形式，标明每条的依据），不能直接执行修改。
- 如果方案中存在"是否要一并处理"这类不影响正确性但涉及风格/规范取舍的问题，要作为独立问题列出，等用户明确选择，不能自行决定后再告知。
- 如果调研过程中发现原始材料（评审意见/问题清单）之外的同类问题或遗漏项，同样要在方案阶段一并呈现，而不是执行完后才提及，给用户完整的决策信息。
- 执行完成后，如涉及可编程校验的正确性条件（如"所有必填字段都已配置""无重复映射"），应当用脚本/工具跑一遍复核，将复核结果（通过/不通过+具体项）作为完成汇报的一部分，而不是仅凭改完的文本判断。
```

## 设计说明（为什么是这个结构）

这个模板背后对应的是一套"先查证据、后出方案、留检查点、再动手、再复核"的处理节奏，来自对一次电商配置迁移评审落地任务的复盘：

1. **Context 段解决"Agent 从哪里开始查"的问题**：如果不给已知的偏差来源和参照样例，Agent 会从零假设，浪费大量轮次重新发现"新旧系统字段解析规则不同"这类用户已经知道的背景。
2. **Request 段的关键是"原始材料要给全量，不要转述"**：评审意见这类材料往往需要逐条对应到具体的修改点，转述会丢失评论者的原话和上下文（谁提的、在哪一行提的），转述后 Agent 可能会误解意见的准确指向。
3. **OutputFormat 段要求"原理说明"而不只是"改了什么"**：一次性的操作记录用完就丢，但如果文档里写清楚了"为什么这个配置解析器会在这种情况下失败"，这份文档就能被下一次同类任务复用，减少重复调研成本。
4. **Constraints 段的核心是治理"看似合理但没验证"的风险**：配置驱动型系统（字段通过字符串名字反射查找、默认值走运行时类型转换）最容易出现"字段名读起来很像对的但实际在 proto 里不存在"这类隐蔽问题，必须靠逐字段的源码/清单核对，而不能靠经验判断。同时要求调研和执行分离——调研阶段允许并行、允许犯错重来，执行阶段（改文件）必须收敛到确认后的单一方案再动手。
5. **Checkpoint 段是这套流程里最容易被跳过、但最重要的一环**：如果 Agent 在方案阶段发现了"用户没问但同样存在的问题"（比如多找出两个同类字段），必须把这类发现和"用户明确要求的修改"分开呈现，并且对"要不要一并处理"这种没有唯一正确答案的取舍问题，显式提交给用户选择，而不是自己拍板。修改完成后，如果正确性条件可以用脚本验证（比如"必填字段是否全部覆盖"），必须真的跑一遍，而不是靠读代码后的印象下结论。
